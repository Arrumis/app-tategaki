# **tategaki 実装計画書**

本ドキュメントは「tategaki 仕様書 Ver.4」および「データ構造定義書 Ver.1」に基づき、開発を円滑に進めるための具体的なステップとタスクを定義したものである。

## 1. 開発マイルストーン

プロジェクトを5つのフェーズに分けて進行する。

| フェーズ | 概要 | 重点項目 |
| :--- | :--- | :--- |
| **Phase 1** | プロジェクト基盤の構築 | 環境構築、ディレクトリ構造の決定 |
| **Phase 2** | データアクセス層とAPIの実装 | JSON操作、Expressによるサーバー機能 |
| **Phase 3** | クローラー（Playwright）の実装 | 小説本文・画像の取得・自動巡回 |
| **Phase 4** | UI/UX（フロントエンド）の実装 | 美しい縦書きビューワー、レスポンシブデザイン |
| **Phase 5** | セキュリティ・最適化・Docker化 | Basic認証、ログ管理、環境統合 |

---

## 2. 詳細タスクリスト

### Phase 1: プロジェクト基盤の構築
- [ ] ディレクトリ構造の初期化 (`data/`, `src/`, `public/`)
- [ ] `package.json` の作成と依存関係のインストール
    - `express`, `playwright`, `dotenv`, `cors`, `morgan` 等
- [ ] `.gitignore` の設定（`data/` 内の実データや `.env` を除外）
- [ ] Docker初期設定 (`Dockerfile`, `docker-compose.yml` のプロトタイプ)

### Phase 2: データアクセス層とAPIの実装
- [ ] 定数・設定管理の実装 (`src/config.js`)
- [ ] ファイルシステム（JSON）操作ユーティリティの実装 (`src/lib/storage.js`)
    - 小説情報の読み書き
    - お気に入りリストの更新
    - しおり（bookmark）の保存
- [ ] Express Route の実装 (`src/routes/api.js`)
    - `GET /api/favs/:id`: お気に入り一覧取得
    - `GET /api/novels/:site/:id/info`: 小説詳細取得
    - `GET /api/novels/:site/:id/ep/:no`: 本文取得
    - `POST /api/bookmark`: しおり更新

### Phase 3: クローラー（Playwright）の実装
- [ ] サイト別スクレイピングロジックの実装 (`src/crawler/sites/`)
    - `narou.js`, `kakuyomu.js`
- [ ] 画像ダウンローダーの実装
- [ ] 定期巡回スケジューラーの実装 (`src/crawler/scheduler.js`)
    - 3時間おきの実行制御
    - 負荷軽減のためのスリープ処理 (3〜5秒)
- [ ] エラーハンドリングとログ出力

### Phase 4: UI/UX（フロントエンド）の実装
- [ ] デザインシステムの定義 (`public/css/style.css`)
    - HSLカラーパレット、フォント（游明朝）、縦書きユーティリティ
- [ ] コア・ビューワーの実装 (`public/reader.html`, `public/js/viewer.js`)
    - `writing-mode: vertical-rl` の適用
    - スクロールスナップ設定
    - タップエリア判定（右：次、左：前、中央：メニュー）
- [ ] 画面各部の実装
    - ホーム画面（ID入力・リスト表示）
    - 目次画面（チャプターリスト）
    - 設定画面（フォントサイズ、テーマ切り替え）

### Phase 5: セキュリティ・最適化・Docker化
- [ ] Basic認証の導入 (Middleware)
- [ ] 環境変数 (`.env`) によるアクセス設定
- [ ] ログローテーションの設定
- [ ] Dockerイメージの最適化とビルド確認

## 3. 開発優先順位 (マネジメント観点)

1.  **最優先 (P0)**: クローラーによるデータ保全。実データがあればフロントエンド開発もスムーズに進むため、先にクローラーを安定させる。
2.  **優先 (P1)**: 縦書きビューワーの心地よさ。データが揃った状態で実際の読み心地を調整する。
3.  **次点 (P2)**: 複数ID管理。まずは単一IDで完成させ、後にマルチリスト対応を確実に行う。

## 4. リスクと対策
- **サイト仕様変更**: 抽象化したスクレイパーインターフェースを用意し、変更時の修正箇所を最小限にする。
- **データ不整合**: ファイル書き込み時に一時ファイルを使用し、アトミックな書き換えを心がける（破損防止）。
- **著作権/規約**: クローリング間隔を空け、サーバーへ負荷をかけない設計を徹底する（仕様通り3〜5秒）。
